# 面试一问一答 —— 每个项目的核心原理

> 被问到"你这个项目怎么做的"——用一句话讲清原理，再展开一句讲实现。

---

## 项目 1-2：API 调用 + 多轮对话

**问：AI 怎么接收和回复消息的？**
> 通过 HTTP 请求调用 DeepSeek 的 API，把用户消息放在 messages 列表里发给模型，模型返回的消息也在同一个列表里——这就是上下文记忆。

**追问：多轮对话为什么 AI 能记住前面说的？**
> 每次请求都把整段对话历史放进 messages 数组一起发给 API。AI 本身没有记忆，是我们在代码里帮它"记住"的。

---

## 项目 3-4：工具调用（Function Calling）

**问：AI 怎么调用工具？**
> 在 API 请求里加上 tools 参数，告诉模型"你可以用这些函数"。模型想用工具时会在 response 里返回 tool_calls，写清楚函数名和参数。我们代码里根据这个去执行真正的函数，然后把结果传回去。

**追问：AI 怎么知道该用哪个工具？**
> 每个工具都有 name + description + JSON Schema 描述参数。模型根据用户问题和工具描述做判断——就像你告诉一个人"你可以用计算器"和"你可以查天气"，他自然知道该用哪个。

---

## 项目 5-6：文件读写 + 网页搜索

**问：AI 怎么读文件？**
> 给 AI 一个 read_file 工具，参数是文件路径。AI 决定读文件时，代码用 Python 的 open() 打开文件，把内容作为工具结果返回给 AI。

**追问：文件很大怎么办？**
> 限制返回字符数（比如 2000 字），或者做分段检索。这就是 RAG 的前身。

---

## 项目 7-8：Gradio 网页 + 天气查询

**问：你的 AI 助手怎么变成网页的？**
> 用 Gradio 的 ChatInterface——一个专门做聊天界面的 Python 库。把 Agent 的 chat 函数传给 ChatInterface，Gradio 自动生成前端网页，还能在手机上打开。

**追问：天气怎么查的？**
> 用 wttr.in 的免费天气 API，发 HTTP GET 请求拿到 JSON 数据，提取温度和天气状况返回给用户。

---

## 项目 9：文档检索 RAG

**问：什么是 RAG？**
> 检索增强生成。先搜索相关文档片段，再把片段和问题一起发给 AI——让 AI 基于真实资料回答，而不是瞎编。

**追问：为什么不能直接把整个文档发给 AI？**
> 两个原因：文档太长会超模型 token 上限（比如一本书几百万字）；不相关内容会干扰 AI，让它抓不住重点。

---

## 项目 10：TF-IDF 向量检索

**问：怎么从文档里找到最相关的那段？**
> 把文档切成小段，每段用 TF-IDF 算出一个向量。用户问题也转成向量，计算余弦相似度——分数越高的段落越相关。

**追问：TF-IDF 有什么缺点？**
> 只看字面匹配。"苹果手机"和"iPhone"在 TF-IDF 眼里完全无关，因为没有一个字一样。

---

## 项目 13：流式输出

**问：打字机效果怎么做的？**
> API 调用时设 stream=True，服务器不再一次性返回全部结果，而是逐块推送。代码里用 for chunk in stream 拿到每一块文字，yield 出去或 print 出来——用户就看到文字一个个出现了。

**追问：流式和工具调用怎么配合？**
> 分两阶段：先非流式检测要不要调工具（工具调用必须知道完整的 tool_calls），工具执行完后再流式输出最终回答。

---

## 项目 14：Embedding 语义搜索

**问：TF-IDF 和 Embedding 什么区别？**
> TF-IDF 统计词频，Embedding 用神经网络把文字变成向量。Embedding 能理解语义——"苹果手机"和"iPhone"的向量很近，TF-IDF 完全无关。

**追问：Embedding 模型怎么工作的？**
> nomic-embed-text 是一个专门训练来理解语义的神经网络。它不看字面，看上下文——"今天天气真好"和"今天气候不错"在上下文里很相似，所以向量接近。

---

## 项目 15：多 Agent 协作

**问：多个 Agent 怎么协作？**
> 有个 Coordinator（协调员），它分析用户需求后决定派谁干活。比如"计算还剩多少时间"→ 协调员先派研究员查当前时间，再派程序员算剩余百分比，最后汇总回答。

**追问：为什么不一个 Agent 干所有事？**
> 分工让每个 Agent 专注自己领域。研究员不需要懂 Python，程序员不需要会数据分析——各司其职，结果更可靠。

---

## 项目 16：LangChain/LangGraph 框架

**问：LangChain 替代了你手写的什么？**
> 三样：`ChatOpenAI` 替代了 `client.chat.completions`，`@tool` 装饰器替代了 `TOOL_REGISTRY` 字典，`StateGraph` 替代了手动 while 循环做工具调用。

**追问：框架和手写哪个好？**
> 框架省代码、有最佳实践、社区维护。手写让你理解底层——面试能答出来"原理"。我两个都做过，能按需选。

---

## 项目 17：混合检索（BM25 + Embedding + RRF）

**问：为什么混合检索比单路好？**
> BM25 抓精准关键词（比如"Transformer"），Embedding 抓语义（比如"大模型"≈"LLM"）。RRF 把两路排名融合，互相补充，谁好谁上。

**追问：RRF 怎么融合的？**
> Reciprocal Rank Fusion：不管分数绝对大小，只看排名。每个文档的 RRF 分数 = 1/(60+BM25排名) + 1/(60+Embedding排名)。比分高者排前面。

---

## 项目 18：MCP 协议

**问：MCP 是什么？解决什么问题？**
> 工具和 Agent 解耦的标准。之前工具写死在 Agent 代码里——新增工具要改代码重启。MCP 把工具放在独立 Server，Agent 通过 JSON-RPC 动态发现——加工具不用动 Agent。

---

## 项目 19：FastAPI 部署

**问：怎么把 Agent 变成可以对外访问的服务？**
> 用 FastAPI 包装——每个功能对应一个 HTTP 端点：`POST /chat` 对话、`POST /chat/stream` 流式、`GET /tools` 工具列表。FastAPI 自动生成 API 文档（/docs），别人打开网页就能试。

---

## 项目 20：三层记忆系统

**问：你的记忆系统怎么设计的？**
> 三层：短期记忆用 token 滑动窗口保留最近对话；长期记忆存 Chroma 向量库，语义检索历史信息；对话摘要——token 超限时自动压缩旧消息，不丢上下文。
